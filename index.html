Two Performance Tiers

BLAS/LAPACK-linked (Haskell, Julia, Python, R, Swift): call hand-optimised vendor routines (Apple Accelerate on this machine) for dense matrix work — typically 5–20× faster than pure implementations on large factorisation benchmarks.

Pure implementations (C++, Go, Rust): Rust/nalgebra, C++/Eigen, Go/gonum implement their own algorithms without calling LAPACK. Slower on BLAS-heavy ops, but competitive on element-wise and algorithm-class workloads.

Notable Findings
  • Swift/Accelerate dominates factorisation ops (Cholesky, LU, QR) thanks to Apple Silicon-tuned Accelerate — even beating Python/NumPy and Julia on several ops.
  • Sort: Julia is fastest; Go and Swift are ~4–5× slower, exposing meaningful differences in standard-library sort implementations at 10M elements.
  • Rust near-zero memory for Cholesky, LU Decomposition, QR Decomposition: nalgebra performs these factorisations in-place on the moved input, so no extra allocation is needed — correct behaviour, not a gap in measurement.
  • Haskell is excluded from FFT (shows —): the vector-fftw library uses CDouble rather than Haskell Double, making a safe, comparable benchmark impractical.
  • Go/gonum scores last: gonum is a pure-Go library with no LAPACK linkage, resulting in the highest scores across the board on dense linear algebra.

Glossary — What Each Benchmark Measures

01
Matrix Multiply Fastest: λ Haskell 7.50 ms
A × B where A, B are 1000×1000

The canonical dense linear algebra kernel. Measures peak BLAS DGEMM throughput. BLAS-linked languages call cblas_dgemm (O(n³) with SIMD/multi-core optimisation); pure implementations use naive or cache-blocked loops. Results vary by 2–10× depending on whether vendor BLAS is used.

02
Matrix Inverse Fastest: 🦅 Swift 2.47 ms
A⁻¹ for a 500×500 symmetric positive-definite (SPD) matrix

Computed via LU factorisation (DGETRF) followed by triangular solve (DGETRI). Matrix inversion is rarely used directly in practice — solving Ax=b with Linear System Solve is numerically preferable — but it is a standard benchmark for full LAPACK integration.

03
LU Decomposition Fastest: 🦅 Swift 1.17 ms
A = P·L·U factorisation of a 500×500 matrix

Partial-pivot LU (DGETRF) is the workhorse factorisation underlying matrix inversion and general linear solves. Measures LAPACK integration and cache efficiency on a medium-sized problem. Often the fastest path for non-symmetric systems.

04
Eigenvalue Decomp Fastest: 🐍 Python 6.03 ms
Full symmetric eigendecomposition of a 300×300 matrix

Computes all eigenvalues and eigenvectors of a symmetric matrix (DSYEV/DSYEVD). Used in PCA, graph algorithms, quantum mechanics, and structural analysis. Iterative reduction to tridiagonal form makes this slower than Cholesky but faster than general (non-symmetric) eigen for the same size.

05
Cholesky Fastest: 🦅 Swift 0.33 ms
L·Lᵀ factorisation of a 500×500 SPD matrix

The fastest dense factorisation for symmetric positive-definite matrices (DPOTRF). Half the cost of LU due to symmetry exploitation. Used in Kalman filters, Gaussian processes, and least-squares problems. A sensitive indicator of raw BLAS Level-3 throughput.

06
SVD Fastest: 🦅 Swift 14.68 ms
Full thin SVD of a 500×300 matrix (U, Σ, Vᵀ)

Singular Value Decomposition (DGESDD) is the foundation of dimensionality reduction (PCA/LSA), pseudo-inverse computation, and low-rank approximation. Uses a divide-and-conquer algorithm internally. More expensive than Cholesky or LU due to iterative bidiagonalisation.

07
Linear System Solve Fastest: 🦅 Swift 3.29 ms
Solve Ax = b for 1000×1000 SPD A and 1000-element b

The practical alternative to matrix inversion: factorises A (Cholesky for SPD or LU for general) then performs triangular solves. The correct numerical approach for Ax=b. Larger problem size (1000) than the factorisation benchmarks to stress memory bandwidth alongside compute.

08
Vector Dot Product Fastest: ∑ Julia 2.87 ms
x·y for two 10-million-element vectors

A memory-bandwidth-bound BLAS Level-1 operation (DDOT). At 10M doubles the vectors exceed typical L3 cache, so performance reflects RAM bandwidth and SIMD utilisation rather than arithmetic throughput. Exposes overhead from interpreted dispatch (R, Python) vs compiled tight loops.

09
Hadamard Product Fastest: ∑ Julia 0.71 ms
C = A ⊙ B + C for three 1000×1000 matrices

Element-wise (pointwise) multiply-then-add — not a BLAS operation, so every language must loop over 1M doubles itself. Tests compiler vectorisation (SIMD auto-vectorisation in Rust/C++/Julia) and interpreter overhead (Python, R). Julia and Rust typically win here.

10
QR Decomposition Fastest: 🦅 Swift 2.56 ms
Householder QR of a 500×500 matrix (R factor)

QR factorisation (DGEQRF) underpins least-squares solvers, eigenvalue algorithms, and orthogonalisation. Only the compact R factor is forced in all languages; forming the full Q is significantly more expensive. Haskell in particular would be ~400× slower if Q were materialised via its Haskell-level Householder products.

11
FFT (real, 1M) Fastest: 🐍 Python 8.11 ms
Real FFT of 2²⁰ = 1 048 576 samples

Fast Fourier Transform (O(n log n)) is the backbone of signal processing, audio/image analysis, spectral methods, and fast polynomial multiplication. Completely different algorithm class from BLAS. Implementations range from FFTW (near-optimal, used by Python/NumPy and Julia/FFTW.jl) to RustFFT (pure-Rust) and gonum's pure-Go FFT. Haskell is excluded (N/A).

12
Sort 10M floats Fastest: ∑ Julia 148.17 ms
Unstable sort of 10 million random float64 values

A pure-algorithm, comparison-based sort workload with no linear-algebra library involvement. Tests the quality of each language's standard-library sort (introsort, pdqsort, radix sort). At 10M elements the working set exceeds L3 cache, making memory-access pattern and branch-prediction efficiency critical. Results span a 7× range across languages.

Appendix — Changelog

  • 2026-03-26 fix Haskell — LU Decomposition benchmark corrected
    A pull-request from @jonocarroll switched the LU harness to luPacked + evaluate. However, GHC's evaluate only forces the outermost constructor to WHNF; the underlying LAPACK dgetrf call remained a thunk, producing spurious 0.00 ms / 0.00 MB results. Fixed by forcing the explicit L and U factor matrices (lu m with forceM l >> forceM u), which drives evaluation all the way through. Corrected timings: 17.29 ms, 130.36 MB. Thanks @jonocarroll for the contribution!
